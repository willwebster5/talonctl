# Detection Chaining: Correlating Rule Triggers

This document covers how to create behavioral rules that reference detections generated by other rules, enabling compound detections and detection-based attack chains.

## Overview

Detection chaining allows you to:
- **Combine multiple rule triggers** into a single high-fidelity alert
- **Create compound detections** from correlated alerts
- **Build attack chain detections** using rule outputs as inputs
- **Reduce alert fatigue** by requiring multiple indicators

## Understanding Rule Trigger Events (RTEs)

When a rule fires, it generates a Rule Trigger Event (RTE) that can be queried:

### RTE Location and Fields

```cql
#repo="xdr_indicatorsrepo"
Ngsiem.event.type="ngsiem-rule-trigger-event"
```

| Field | Description |
|-------|-------------|
| `Ngsiem.event.type` | Always `ngsiem-rule-trigger-event` |
| `Ngsiem.event.outcome` | `correlation-rule-detection`, `behavioral-detection`, `behavioral-case` |
| `Ngsiem.part.query.{name}` | Event IDs for each named query in correlate() |
| `rule.id` | Unique rule identifier |
| `rule.name` | Rule display name |

### Querying RTEs

```cql
// All correlation rule detections
#repo="xdr_indicatorsrepo"
Ngsiem.event.type="ngsiem-rule-trigger-event"
Ngsiem.event.outcome="correlation-rule-detection"

// All behavioral detections
#repo="xdr_indicatorsrepo"
Ngsiem.event.type="ngsiem-rule-trigger-event"
Ngsiem.event.outcome="behavioral-detection"

// Specific rule by name
#repo="xdr_indicatorsrepo"
Ngsiem.event.type="ngsiem-rule-trigger-event"
rule.name="Microsoft - Entra ID - Potential Successful Brute Force Attack"
```

## Pattern 1: Correlate Multiple Rule Triggers

Combine alerts from different rules when they affect the same user:

```cql
correlate(
  BruteForceAlert: {
    #repo="xdr_indicatorsrepo"
    Ngsiem.event.type="ngsiem-rule-trigger-event"
    Ngsiem.event.outcome="correlation-rule-detection"
    rule.name=/Brute Force|Password Spray/i
  },
  PrivilegeEscalationAlert: {
    #repo="xdr_indicatorsrepo"
    Ngsiem.event.type="ngsiem-rule-trigger-event"
    rule.name=/Privilege Escalation|Admin Role/i
    | user.name <=> BruteForceAlert.user.name
  },
  within=4h,
  globalConstraints=[user.name]
)
| _Severity := 90
| _Risk := "Critical"
| _Reason := "Brute force followed by privilege escalation"
```

## Pattern 2: Cross-Vendor Alert Correlation

Correlate alerts from different security vendors/sources:

```cql
correlate(
  EndpointAlert: {
    #repo="xdr_indicatorsrepo"
    Ngsiem.event.type="ngsiem-rule-trigger-event"
    rule.name=/Endpoint|EDR|Malware/i
  },
  CloudAlert: {
    #repo="xdr_indicatorsrepo"
    Ngsiem.event.type="ngsiem-rule-trigger-event"
    rule.name=/AWS|Cloud|IAM/i
    | user.email <=> EndpointAlert.user.email
  },
  within=24h,
  globalConstraints=[user.email]
)
| _Severity := 90
| _Reason := "Endpoint and cloud alerts for same user"
```

## Pattern 3: Alert → Raw Event Correlation

Correlate a rule trigger with subsequent raw events:

```cql
correlate(
  InitialAlert: {
    #repo="xdr_indicatorsrepo"
    Ngsiem.event.type="ngsiem-rule-trigger-event"
    rule.name="AWS - Unusual API Activity"
  },
  SubsequentActivity: {
    #Vendor="aws"
    event.action=/Create|Delete|Modify/
    | Vendor.userIdentity.arn <=> InitialAlert.Vendor.userIdentity.arn
  },
  sequence=true,
  within=1h,
  globalConstraints=[Vendor.userIdentity.arn]
)
```

## Pattern 4: Threshold-Based Alert Correlation

Alert when a user triggers multiple different detections:

```cql
correlate(
  Alert1: {
    #repo="xdr_indicatorsrepo"
    Ngsiem.event.type="ngsiem-rule-trigger-event"
  },
  Alert2: {
    #repo="xdr_indicatorsrepo"
    Ngsiem.event.type="ngsiem-rule-trigger-event"
    | user.email <=> Alert1.user.email
    | test(rule.name != Alert1.rule.name)  // Different rule
  },
  Alert3: {
    #repo="xdr_indicatorsrepo"
    Ngsiem.event.type="ngsiem-rule-trigger-event"
    | user.email <=> Alert1.user.email
    | test(rule.name != Alert1.rule.name)
    | test(rule.name != Alert2.rule.name)  // Third different rule
  },
  within=24h,
  globalConstraints=[user.email]
)
| _Severity := 90
| _Reason := "User triggered 3+ different detections"
```

## Pattern 5: CrowdStrike First-Party Detection Correlation

Reference CrowdStrike Detection Summary Events (DSE):

### Available DSE Types

| Event Type | Source |
|------------|--------|
| `EPPDetectionSummaryEvent` | Endpoint Protection Platform |
| `IDPDetectionSummaryEvent` | Identity Protection |
| `XDRDetectionSummaryEvent` | Extended Detection & Response |
| `CloudDetectionSummaryEvent` | Cloud Security |
| `LogScaleDetectionSummaryEvent` | LogScale/Humio |
| `CustomIOADetectionSummaryEvent` | Custom IOA rules |

### Correlating DSE Events

```cql
correlate(
  EndpointDetection: {
    #event_simpleName="EPPDetectionSummaryEvent"
    Tactic=/Persistence|PrivilegeEscalation/
  },
  IdentityDetection: {
    #event_simpleName="IDPDetectionSummaryEvent"
    | UserName <=> EndpointDetection.UserName
  },
  within=4h,
  globalConstraints=[UserName]
)
| _Severity := 90
| _Reason := "Endpoint + Identity detections for same user"
```

## Pattern 7: Alert Escalation Chain

Create severity escalation based on alert patterns:

```cql
correlate(
  LowSeverityAlerts: {
    #repo="xdr_indicatorsrepo"
    Ngsiem.event.type="ngsiem-rule-trigger-event"
    rule.severity=/Low|Informational/i
    | groupBy([user.email], function=[alert_count := count()])
    | test(alert_count >= 5)
  },
  MediumAlert: {
    #repo="xdr_indicatorsrepo"
    Ngsiem.event.type="ngsiem-rule-trigger-event"
    rule.severity=/Medium/i
    | user.email <=> LowSeverityAlerts.user.email
  },
  HighAlert: {
    #repo="xdr_indicatorsrepo"
    Ngsiem.event.type="ngsiem-rule-trigger-event"
    rule.severity=/High|Critical/i
    | user.email <=> LowSeverityAlerts.user.email
  },
  sequence=true,
  within=24h,
  globalConstraints=[user.email]
)
| _Severity := 90
| _Reason := "Alert escalation pattern: low → medium → high"
```

## Complete YAML Template: Alert Correlation Rule

```yaml
name: "Compound Detection - Multi-Alert Correlation"
resource_id: compound_multi_alert_correlation
description: |
  Creates high-fidelity alert when multiple detection rules
  fire for the same user within a time window.

  Alert Correlation Pattern:
  1. Initial detection from brute force or spray rule
  2. Subsequent detection from privilege escalation rule
  3. Both affecting same user within 4 hours

  This pattern indicates successful credential compromise
  followed by attacker privilege escalation.

severity: 90
status: active
tactic: TA0004
technique: T1078
search:
  filter: |
    correlate(
      InitialCompromise: {
        #repo="xdr_indicatorsrepo"
        Ngsiem.event.type="ngsiem-rule-trigger-event"
        Ngsiem.event.outcome="correlation-rule-detection"
        rule.name=/Brute Force|Password Spray|Credential/i
      },
      PrivilegeAbuse: {
        #repo="xdr_indicatorsrepo"
        Ngsiem.event.type="ngsiem-rule-trigger-event"
        rule.name=/Privilege|Escalation|Admin|Role/i
        | user.email <=> InitialCompromise.user.email
      },
      sequence=true,
      within=4h,
      globalConstraints=[user.email]
    )
    | table([
        InitialCompromise.rule.name,
        PrivilegeAbuse.rule.name,
        InitialCompromise.user.email,
        InitialCompromise.@timestamp,
        PrivilegeAbuse.@timestamp
    ])
  lookback: 8h
  trigger_mode: summary
  outcome: detection
operation:
  schedule:
    definition: '@every 1h'
```

## Important Limitations

### Cannot Chain Certain Rule Types

> **Limitation**: Fusion SOAR workflow triggers from behavioral rules cannot be used as part of other behavioral rules.

This means:
- Rules that trigger SOAR workflows cannot have their outputs used in correlate()
- Design rules with chaining in mind if you plan to correlate them

### Time Window Considerations

- RTEs may have slight delays from rule execution
- Set `within` conservatively to account for processing time
- Set `lookback` significantly larger than `within`

### Performance

- Correlating many RTEs can be resource-intensive
- Filter RTEs specifically (by rule name) rather than broadly
- Use `globalConstraints` to narrow correlation scope

## Best Practices

### 1. Use Specific Rule Name Filters

```cql
// Good - specific rules
rule.name="AWS - IAM Privilege Escalation"

// Avoid - too broad
rule.name=*
```

### 2. Always Include Entity Correlation

```cql
// Always correlate on a shared entity
globalConstraints=[user.email]
// or
| user.email <=> OtherAlert.user.email
```

### 3. Document Rule Dependencies

```yaml
description: |
  This rule depends on the following rules:
  - "AWS - Brute Force Detection" (must fire first)
  - "AWS - Privilege Escalation" (must fire second)

  Ensure dependent rules are active and functioning.
```

### 4. Test Rule Chains Incrementally

1. Verify each source rule fires correctly
2. Query RTEs to confirm they're generated
3. Test correlate() with relaxed time windows
4. Tighten parameters once validated
